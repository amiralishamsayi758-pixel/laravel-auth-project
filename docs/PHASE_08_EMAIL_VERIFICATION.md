# Phase 8: Email Verification

## Objective

Phase 8 adds Laravel's standard email-verification contract, registered-event listener, notification, signed URLs, verification request, `Verified` event, and `verified` middleware to the existing manual authentication architecture. No starter kit or third-party authentication package is used.

## User flow

```text
Registration form
    -> development registration-code step
    -> create unverified user
    -> dispatch Registered
    -> standard VerifyEmail notification
    -> authenticate user
    -> verification notice
       -> resend (throttled), or open signed URL
       -> EmailVerificationRequest validates user/id/hash/signature
       -> fulfill()
       -> gmail_verified_at populated + Verified event
       -> verified dashboard
```

The existing preliminary six-digit development-code step remains intact. It no longer marks Gmail as verified; only Laravel's signed email-verification URL does that.

## Architecture

### `MustVerifyEmail` and the Gmail column

`User` implements `Illuminate\Contracts\Auth\MustVerifyEmail` and uses Laravel's `Illuminate\Auth\MustVerifyEmail` trait for notification behavior. The project intentionally uses `gmail` and `gmail_verified_at` instead of Laravel's default `email` and `email_verified_at`, so the model provides compatible implementations:

- `getEmailForVerification()` returns `gmail`.
- `hasVerifiedEmail()` reads `gmail_verified_at`.
- `markEmailAsVerified()` safely timestamps `gmail_verified_at`.

The existing datetime cast, password-reset Gmail routing, fillable fields, hidden fields, hashed password cast, and authentication behavior are preserved.

### Registration and notification

After the existing verification-code controller creates the user, it dispatches Laravel's `Registered` event exactly once. Laravel's framework listener sees that the user implements `MustVerifyEmail` and sends the standard `VerifyEmail` notification. The controller does not send a second notification manually. The user remains logged in and is redirected to `verification.notice`.

### Signed verification URL

The standard notification generates a temporary signed URL named `verification.verify`. It includes the authenticated user's ID and a SHA-1 hash of `getEmailForVerification()`. The route has `auth`, `signed`, and `throttle:6,1` middleware.

`VerifyEmailController` accepts `EmailVerificationRequest`. Laravel therefore validates the signature, authenticated ownership, route ID, and Gmail hash. For an unverified user, `$request->fulfill()` calls the model verification method and dispatches the standard `Verified` event. Already-verified users are redirected idempotently without a second event.

### Verified middleware

The dashboard is protected by both `auth` and `verified`. Guests go to login. Authenticated unverified users go to `verification.notice`. Verified users can access the dashboard normally. The login controller retains `intended()` behavior; middleware performs the verification redirect without duplicating checks in login.

## Routes

| Method | URI | Name | Middleware |
|---|---|---|---|
| GET | `/email/verify` | `verification.notice` | `web`, `auth` |
| GET | `/email/verify/{id}/{hash}` | `verification.verify` | `web`, `auth`, `signed`, `throttle:6,1` |
| POST | `/email/verification-notification` | `verification.send` | `web`, `auth`, `throttle:6,1` |

The dashboard additionally uses `verified`. Resend and logout are POST forms with CSRF tokens.

## Controllers and view

- `EmailVerificationPromptController` redirects verified users to dashboard and shows the notice to unverified users.
- `VerifyEmailController` delegates security validation and fulfillment to Laravel.
- `EmailVerificationNotificationController` avoids resending to verified users and otherwise sends the standard notification.
- `auth/verify-email.blade.php` matches the existing Persian authentication UI, explains inbox/spam checking, displays resend confirmation, and provides CSRF-protected resend and logout forms.

No internal IDs or hashes are rendered by the notice page.

## Local mail testing

`.env.example` already uses the safe `MAIL_MAILER=log` default. With this setting, verification messages are written to `storage/logs/laravel.log` rather than delivered. Set `APP_URL` to the exact local URL used in the browser so signed links have the correct host.

Mailpit may optionally be used as a browser inbox by changing local `.env` mail values to its SMTP host and port (commonly `127.0.0.1:1025`). Production SMTP credentials must be stored only in deployment secrets or an uncommitted `.env`; they must never be committed.

## Security considerations

- Verification uses temporary signed URLs generated by Laravel.
- `EmailVerificationRequest` validates authenticated ownership, ID, and Gmail hash.
- Invalid signatures, hashes, guests, and cross-user attempts are rejected.
- `$request->fulfill()` is used instead of directly trusting route parameters or manually setting the timestamp.
- `Verified` is not manually duplicated.
- Registration dispatches `Registered` once and does not manually duplicate its notification.
- Handler and resend routes are rate-limited.
- Resend and logout require POST plus CSRF.
- Verified users are handled idempotently.

## Manual testing checklist

1. Set `APP_URL` to the Laravel server URL and use `MAIL_MAILER=log` locally.
2. Register and complete the existing six-digit development-code step.
3. Confirm redirect to `/email/verify` and inability to open `/dashboard`.
4. Find the verification URL in `storage/logs/laravel.log` and open it in the same authenticated browser.
5. Confirm the success message and dashboard access.
6. Log out and back in; confirm the verified account reaches dashboard.
7. For resend testing, register another account, use the resend button, and confirm the status message and throttle behavior.
8. Optionally configure Mailpit locally and confirm the link arrives in its browser inbox.

## Common failure cases

- **Email not sent:** ensure `Registered` is dispatched after creation and the user implements `MustVerifyEmail`.
- **Verification route missing:** confirm the exact `verification.verify` route name exists.
- **Signed URL invalid:** ensure `APP_URL`, scheme, host, port, and system time match the browser environment.
- **Mail configuration incorrect:** use `MAIL_MAILER=log` first, then inspect Laravel logs.
- **User already verified:** the notice and resend endpoints redirect to dashboard; handler is idempotent.
- **Redirect loop:** keep the notice routes on `auth`, and apply `verified` only to protected destinations.
- **Notification sent twice:** dispatch `Registered` once and do not also call the notification manually during registration.
